# Codebase Concerns

> Analyzed 2026-03-22. This is a fresh Next.js 16 + React 19 scaffolding project â€” most concerns are **potential risks** that will materialize as the codebase grows. Actively address the critical ones before adding meaningful application code.

---

## Critical

### C-1: No Security Headers Configured
**File:** `next.config.ts`

The `NextConfig` is completely empty. There are no security-related headers:
- No `X-Frame-Options` / `frame-ancestors` CSP directive â†’ clickjacking risk
- No `X-Content-Type-Options: nosniff` â†’ MIME-type sniffing risk
- No `Referrer-Policy` â†’ referrer leakage risk
- No `Permissions-Policy` â†’ unused browser feature exposure
- No CSP (Content Security Policy) â†’ XSS vector widened

**Action:** Add a `headers()` function to `next.config.ts` with a comprehensive security header policy before any user-facing deployment.

---

### C-2: Missing Favicon
**File:** `src/app/favicon.ico`

`favicon.ico` is listed as present in `src/app/` but the `public/` directory (where Next.js expects static assets) has no favicon. Without a favicon:
- Browser tabs show default browser icon
- No brand identity
- Potential 404s in server logs on every page load

**Action:** Place a favicon in `app/` (using the App Router convention) or `public/` and update `metadata` in `layout.tsx` accordingly.

---

### C-3: Generic / Unchanged Metadata
**File:** `src/app/layout.tsx`

```tsx
title: "Create Next App"
description: "Generated by create next app"
```

Default create-next-app metadata is still in place. This is the canonical "this app was never customized" signal and is a P0 for any deployment.

**Action:** Replace with real application metadata immediately.

## High

### H-1: No Error Boundary
**Files:** `src/app/layout.tsx`, `src/app/page.tsx`

There is no `error.tsx` (Next.js App Router error boundary) and no React error boundary anywhere. Any unhandled runtime error will result in a full-page crash with no recovery option for the user.

**Action:** Create `src/app/error.tsx` as a minimum. Add a global `GlobalError` component for root-level crashes.

---

### H-2: No Testing Setup
**Files:** `package.json`

Zero testing dependencies (`@testing-library/*`, `vitest`, `jest`, `playwright`, etc.) are present. No test files exist. For a project expected to grow, this is a compounding risk â€” untested code becomes unchangeable code.

**Action:** Add a test runner and at minimum one smoke test for the root layout before adding meaningful business logic.

---

### H-3: External Links Lack Accessibility Labels
**File:** `src/app/page.tsx`

```tsx
<a href="https://vercel.com/templates?..." ...>
  Templates
</a>
```

All external navigation links use generic text content. Users relying on screen readers will not understand the destination. The `rel="noopener noreferrer"` is correctly set, but `aria-label` or visually hidden descriptive text is missing.

**Action:** Add descriptive `aria-label` attributes or SR-only text to all external links.

---

### H-4: No Environment Configuration Scaffolding
**Files:** `package.json`, root directory

There is no `.env.local`, `.env.example`, or `.env.production` file. As the project grows, secrets will be added ad-hoc, increasing the risk of accidental secret committed to version control.

**Action:** Create `.env.example` immediately documenting all expected env vars, and add a pre-commit hook or `eslint-plugin-simple-import-sort` / `git secrets` check to prevent `.env.local` commits.

---

## Medium

### M-1: Next.js 16 Is Bleeding Edge â€” Docs May Lag
**File:** `package.json`

```json
"next": "16.2.1"
```

Next.js 16 is far ahead of stable training data. The project notes (`AGENTS.md`) correctly instruct reading `node_modules/next/dist/docs/` before writing code, but this guidance is not enforced programmatically. Breaking API changes could be introduced without notice.

**Action:** Pin exact versions in `package.json` (`next@16.2.1` is already pinned; ensure `react` and `react-dom` are also pinned exactly, not version-ranged). Create a `docs-check` npm script that runs before `build`.

---

### M-2: Tailwind v4 CSS Variable Strategy Conflict
**Files:** `src/app/globals.css`, `postcss.config.mjs`

`globals.css` defines `:root { --background: #ffffff; --foreground: #171717; }` and references them inside `@theme inline {}`. This is Tailwind v4's new `@theme` CSS-variable-first approach. However:

- The `html` element in `layout.tsx` has a hardcoded `className="... h-full antialiased"` with **no `dark` class** toggled â€” meaning the `@media (prefers-color-scheme: dark)` block in `globals.css` will only apply if the OS preference is set, not via a class toggle.
- Tailwind v4 uses `class="dark"` strategy for manual dark mode toggling, but the CSS defines a `@media` strategy only. The current setup is inconsistent.
- `postcss.config.mjs` uses `@tailwindcss/postcss` (correct for v4), but there is no `tailwind.config.ts` â€” v4 uses `@theme` in CSS instead, which is fine but non-obvious.

**Action:** Decide on a dark mode strategy (class vs. media) and be consistent. Document it. If using class-based dark mode, add a theme toggle component.

---

### M-3: Font Loading Without `display` Parameter
**File:** `src/app/layout.tsx`

```tsx
Geist({ variable: "--font-geist-sans", subsets: ["latin"] })
```

`next/font/google` defaults to `display: 'swap'`, but the behavior is more predictable when explicitly set. Additionally, both `Geist` and `Geist_Mono` are loaded even if only one is used in `globals.css`.

**Action:** Explicitly set `display: 'swap'` (or another strategy) in font configs. Audit which fonts are actually used.

---

### M-4: TypeScript Target Below Modern Standard
**File:** `tsconfig.json`

```json
"target": "ES2017"
```

ES2017 is from 2017. Modern Next.js + React 19 projects benefit significantly from ES2022+ features (`Array.findLast`, `cause` on errors, top-level await in modules, etc.). Using an older target means TypeScript may desugar syntax that could otherwise be native.

**Action:** Update `target` to `ES2022` or `"ESNext"`. This is low-risk since `moduleResolution: "bundler"` + Next.js's SWC compiler handles everything.

---

### M-5: No Prettier Configuration
**Files:** root directory

ESLint is configured but Prettier is not. Without a shared Prettier config, code style will diverge across contributors, creating noise in diffs and reducing review quality.

**Action:** Add `.prettierrc` and `.prettierignore`. Integrate with ESLint via `eslint-config-prettier`.

---

### M-6: No CI/CD Pipeline
**Files:** root directory

No GitHub Actions, no Vercel deploy previews configured. Bugs and type errors can reach `main` unchallenged.

**Action:** Add a minimal GitHub Actions workflow that runs `npm run build` and `npm run lint` on PRs.

---

### M-7: No Husky / Commitlint
**Files:** root directory

No pre-commit hooks. Malformed commits (missing types, lint errors) can reach the repo.

**Action:** Add `husky` + `commitlint` with a conventional commit config.

---

## Low / Informational

### L-1: This Is a Fresh Scaffold â€” Most of the Codebase Is Boilerplate
The `page.tsx` is the unmodified `create-next-app` template. All `CONCERNS` above are **pre-emptive**; the actual application has not yet introduced domain-specific bugs, data flows, or security surfaces. This document should be revisited once meaningful features are added.

### L-2: No Source Maps for Production Builds
`tsconfig.json` has `noEmit: true` (correct for Next.js), but there is no explicit production source map configuration. For debugging and error tracking, consider enabling source maps in `next.config.ts` via `productionBrowserSourceMaps: true`.

### L-3: `ESLint` Version
```json
"eslint": "^9"
```

ESLint v9 uses a flat config format (which the project correctly uses in `eslint.config.mjs`). However, many popular ESLint plugins still do not support ESLint v9 flat config. This will become a compatibility problem as more ESLint plugins are added.

### L-4: No API Layer or Data Layer Defined
The project has no `app/api/`, `lib/`, `services/`, or database configuration of any kind. All data concerns are deferred â€” but this also means there are no current concerns around ORM usage, SQL injection, missing authentication, or data serialization. Track these as they emerge.

### L-5: `flex-1` on `<main>` Without Explicit Parent Constraints
In `page.tsx`:
```tsx
<div className="flex flex-col flex-1 items-center justify-center ...">
  <main className="flex flex-1 ...">
```
The `flex-1` on `<main>` is meant to fill available space, but the parent `<div>` has no explicit `height` or `min-height`. On very short viewports the content may collapse or overflow unexpectedly.

### L-6: `@media (prefers-color-scheme: dark)` in CSS â€” Browser Support
`prefers-color-scheme` is widely supported, but not universal (older mobile browsers, some e-ink devices). The current dark mode implementation silently fails on unsupported browsers, showing the dark theme colors by default. Consider a JS-based fallback.
